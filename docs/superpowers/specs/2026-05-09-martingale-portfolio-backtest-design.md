## 1. 背景与目标

用户希望在当前 Grid Binance 平台上新增一套独立的“混合马丁组合回测与实盘发布系统”。该系统用于研究、回测、筛选并半自动发布马丁倍投网格策略。

马丁策略定义为：按一定价格间隔逆向加仓，且每层下单金额按倍率或序列递增。例如 BTCUSDT 合约首单 10 USDT，价格逆向移动 1% 后加仓 20 USDT，再逆向 1% 后加仓 40 USDT，以此类推；整个仓位按加权均价整体止盈，例如整体盈利 1% 后平仓。

“混合”表示系统必须支持多币种、多金额模型、多间隔模型、多价格区间、多方向、多策略同时组合运行。用户希望通过本地硬件，包括 5090 显卡和 9950X CPU，执行大规模随机搜索与智能搜索回测，筛出风险收益较优的组合包，并在人工确认后应用到实盘。

本系统的工程目标是实现完整闭环：配置参数空间、读取历史数据、执行两阶段回测、防过拟合评估、生成 Portfolio 候选、展示风险摘要、人工确认、创建并启动实盘组合、运行中风控与恢复。

本系统不承诺稳定盈利或无风险。系统必须明确展示马丁策略的尾部风险、最大资金占用、最大可能亏损、强平风险和回测不等于未来收益的警告。

## 2. 范围

### 2.1 必须实现

- 新增独立策略类型 `martingale_grid`。
- 支持现货 Spot 与 Binance USDT-M Futures。
- 支持 Binance 全部 USDT 交易对，并允许用户通过白名单、黑名单和筛选规则选择范围。
- 支持 Long only、Short only、Long + Short 双向独立运行。
- 合约 Long + Short 同步运行时只支持 Hedge Mode。
- 合约保证金模式默认逐仓 `isolated`，高级模式允许全仓 `cross`。
- 合约杠杆参与参数搜索，但实盘发布前必须人工确认。
- 同一合约 symbol 同时运行多个策略时，必须使用相同 `margin_mode` 与 `leverage`，冲突时禁止发布。
- 支持两阶段回测：K 线海选 + aggTrades/成交级精测 Top N。
- 支持随机搜索与智能搜索。
- 支持生存优先筛选与自定义评分权重。
- 支持混合资金池：全局预算、单币种预算、单方向预算、单策略预算。
- 支持组合级、币种级、方向级、策略级止损与熔断。
- 支持独立 Worker 进程执行重型回测任务。
- 支持专用回测 UI 页面，包含向导模式与专业控制台模式。
- 支持半自动实盘发布：系统生成 Portfolio 候选，用户查看风险摘要后确认启动。
- 支持 Portfolio 整体停止、暂停新开仓、单策略暂停/恢复/停止。
- 支持保守异常恢复：重启后先同步交易所状态，不明确归属的订单标记为 orphan 并暂停相关策略。
- 使用外部行情 SQLite 数据库时必须只读，不允许写入、迁移、VACUUM、checkpoint、建索引。

### 2.2 不在第一版承诺范围

- 不承诺自动盈利。
- 不做完全无人值守的自动轮换实盘策略。
- 不默认使用全仓作为马丁策略的安全模式。
- 不支持 COIN-M Futures 作为第一优先级。
- 不把现货杠杆账户混入第一版。
- 不强制使用 GPU；第一版可以先用 CPU 多线程实现稳定可复现搜索，GPU 加速作为后续优化空间。

## 4. 策略模型

### 4.1 Portfolio

Portfolio 是本系统的实盘发布对象。一个 Portfolio 包含多个 Martingale Strategy Instance，每个实例可以对应不同 symbol、方向、间隔模型、金额模型、止盈止损模型和指标过滤规则。

Portfolio 必须包含：

- 全局预算。
- 单币种预算。
- 单方向预算。
- 单策略预算。
- 最大同时活跃策略数。
- 最大同时活跃 cycle 数。
- 全局止损与熔断规则。
- 搜索来源任务 ID。
- 回测数据范围。
- 评分权重。
- 随机种子。
- 代码版本或配置版本。
- 风险摘要。

### 4.2 Strategy Instance

Strategy Instance 是 Portfolio 内部的单个策略实例。它必须独立记录：

- `strategy_instance_id`。
- symbol。
- market: spot 或 USDT-M futures。
- direction: long 或 short。
- margin mode 与 leverage，仅合约适用。
- entry trigger。
- spacing model。
- sizing model。
- take-profit model。
- stop-loss model。
- indicator rules。
- active cycles。
- 独立资金曲线与风控事件。

### 4.3 Cycle 与 Leg

Cycle 表示从首单开始，到整体止盈或止损结束的一轮马丁过程。Leg 表示首单或某一层补仓。

每个 cycle 必须记录：

- `cycle_id`。
- direction。
- anchor price。
- weighted average entry price。
- open quantity。
- realized pnl。
- unrealized pnl。
- leg 列表。
- 当前止盈/止损状态。

每个 leg 必须记录：

- `leg_index`。
- trigger price。
- notional。
- quantity。
- fee。
- slippage。
- order id。
- fill status。

## 5. 马丁参数能力

### 5.1 开仓触发

系统必须支持组合触发：

- 立即开仓。
- 指标条件触发。
- 价格进入区间触发。
- 时间窗口触发。
- 冷却时间触发。
- 容量限制触发。
- 按方向触发。

示例：BTCUSDT 只有在 RSI < 35、ADX < 25、价格处于指定区间、且该 symbol 活跃策略数未超限时，才允许开 long cycle。

### 5.2 补仓间隔模型

系统必须支持：

- 固定百分比间隔，例如每 1% 一层。
- 倍率间隔，例如 1%、1.5%、2.25%。
- ATR 动态间隔，例如每 1.2 * ATR 一层。
- 自定义序列，例如 0.5%、0.8%、1.3%、2.1%。
- 混合模型，例如前几层固定，后几层倍率放宽，波动高时切 ATR。

### 5.3 下单金额模型

系统必须支持：

- 固定倍投，例如 10、20、40、80。
- 温和倍投，例如倍率 1.2x、1.5x、2x。
- 自定义金额序列，例如 10、20、30、50、80。
- 预算约束自动缩放。
- 超出预算或低于交易所最小下单额时淘汰候选。

### 5.4 整体止盈模型

系统必须支持：

- 固定收益率止盈，例如整体仓位加权均价上方 1%。
- 固定金额止盈，例如盈利 20 USDT。
- ATR 动态止盈。
- 移动止盈。
- 混合止盈，例如达到固定收益率后启动移动止盈。

止盈必须基于 cycle 的整体仓位，而不是每个 leg 独立止盈。

### 5.5 止损与熔断模型

系统必须支持：

- 区间止损。
- ATR 止损。
- 指标止损。
- 单策略回撤百分比止损。
- 单币种回撤金额止损。
- 单方向回撤止损。
- 全局组合回撤金额止损。
- 强平风险提前熔断。

优先级必须是：全局风控 > 币种/方向风控 > 单策略风控 > cycle 级止盈止损。

止损后处理采用组合控制：单策略止损默认暂停该策略；可配置冷却后重启。组合级止损可暂停全部新开仓，并可选择是否强制平掉全部策略。

## 7. 回测系统

### 7.1 两阶段回测

第一阶段是 K 线海选。它使用 OHLCV 快速模拟大量参数组合，目标是筛掉明显不合格候选。K 线模拟必须采用保守的 intrabar 触发顺序，不能乐观假设先止盈后补仓。

第二阶段是成交级精测。它只对 Top N 候选执行更细粒度回放，使用 aggTrades 或等价成交数据模拟更真实的触发、滑点、手续费、资金占用与止损顺序。

### 7.2 时间切分与防过拟合

系统必须支持：

- 手动训练/验证/测试区间。
- Walk-forward 滚动窗口。
- 压力场景库。

压力场景包括：

- 暴跌。
- 暴涨。
- 高波动震荡。
- 低波动横盘。
- 长单边趋势。
- 插针行情。

候选策略必须展示训练段、验证段、测试段、压力段表现。如果候选只在训练段表现优秀，而在验证或压力段失败，必须标记过拟合风险。

### 7.3 搜索方式

系统必须支持随机搜索与智能搜索。

随机搜索要求：

- 参数范围由用户配置。
- 采样数量受配额限制。
- 保存 random seed。
- 同一数据、同一 seed、同一代码版本应可复现。

智能搜索要求：

- 初始随机探索。
- 根据生存优先评分保留 Top 分位。
- 在优秀参数附近变异或收缩范围。
- 多轮迭代。
- 受最大轮数、最大候选数、最长运行时间、取消信号限制。

### 7.4 评价指标

系统默认采用生存优先，然后按自定义权重评分。

生存优先先淘汰：

- 爆仓或强平候选。
- 超过全局回撤限制候选。
- 超过单策略止损限制候选。
- 最大资金占用超预算候选。
- 止损次数过高候选。
- 数据完整度不足候选。

可评分指标包括：

- 总收益。
- 年化收益。
- 最大回撤。
- Calmar。
- Sharpe。
- Sortino。
- 胜率。
- 止损频率。
- 最长浮亏时间。
- 资金利用率。
- 交易次数稳定性。
- 强平距离。

- 现货。

- 固定最小滑点。

## 14. 实盘发布与运行

发布对象是 Portfolio，不是单策略。

发布流程：

1. 回测任务完成。
2. 用户查看候选 Portfolio。
3. 系统展示风险摘要。
4. 用户点击发布意图。
5. 系统检查交易所状态、权限、配额、symbol 冲突、Hedge Mode、margin mode、leverage。
6. 系统生成 pending Portfolio。
7. 用户二次确认。
8. 系统创建 live strategy instances。
9. Worker 或 trading engine 启动运行。

实盘运行必须支持：

- Portfolio 启动。
- 暂停新开仓。
- 整体停止。
- 单策略暂停。
- 单策略恢复。
- 单策略停止。
- 风控事件记录。
- orphan order 提示。

## 15. Binance 合约约束

USDT-M Futures 支持通过 API 修改 margin type 和 leverage，但系统不得静默修改。

发布前必须检查：

- 当前 position mode 是否 Hedge Mode。
- 当前 symbol margin type。
- 当前 symbol leverage。
- 当前是否有持仓。
- 当前是否有挂单。

如果配置不一致：

- 无持仓且无挂单时，可以在用户确认后通过 API 切换。
- 有持仓或挂单时，拒绝自动切换，要求人工处理。

同一 futures symbol 的 margin mode 与 leverage 是 symbol 级约束。系统必须阻止 BTCUSDT 同时运行 `isolated 3x` 与 `cross 5x` 这类冲突策略。

## 16. 统计系统

因为同一 symbol 可以运行多个策略，统计系统必须重新设计为三层：

### 16.1 Strategy Instance 层

统计单个策略实例：

- 收益。
- 回撤。
- active cycles。
- leg 分布。
- 止盈次数。
- 止损次数。
- 资金占用。
- 异常事件。

### 16.2 Symbol Portfolio 层

统计同一 symbol 下所有策略：

- 总敞口。
- 净敞口。
- 多头敞口。
- 空头敞口。
- symbol 级资金占用。
- symbol 级回撤。
- 同 symbol 策略相关风险。

### 16.3 Global Portfolio 层

统计整个 Portfolio：

- 全局资金曲线。
- 全局最大回撤。
- 全局资金占用。
- 全局止损事件。
- 组合级收益风险指标。
- 触发熔断状态。

成交与订单归因必须依赖 `strategy_instance_id`、`cycle_id`、`leg_index`、`direction`，不能只按 symbol 聚合。

---

## 17. 异常恢复

恢复策略采用保守恢复。

系统重启、网络异常或 API 异常后：

1. 暂停新补仓。
2. 同步交易所持仓。
3. 同步挂单。
4. 同步成交历史。
5. 根据 client order id 对账本地 cycle/leg/order 状态。
6. 明确归属的订单继续接管。
7. 不能明确归属的订单标记为 orphan order。
8. 暂停相关策略或 Portfolio。
9. 页面提示人工处理。

系统不能通过 symbol、price、quantity 猜测并自动接管不明确订单。

---

## 18. 安全门禁

以下规则不可绕过：

- 未完成精测和风险摘要的候选不能发布。
- 未人工确认的候选不能启动实盘。
- futures long+short 未确认 Hedge Mode 不能启动。
- 同 symbol margin/leverage 冲突不能启动。
- 超过预算的策略不能启动。
- 数据质量不足的回测结果不能作为优质候选。
- 生存筛选失败的候选不能因为收益高而排到有效候选前面。
- 外部行情数据库不能被写入。
- 不确定归属的订单不能自动接管。

## 19. 成功标准

系统完成后，应满足：

- 用户能在独立回测页面创建混合马丁 Portfolio 回测任务。
- 用户能选择全 USDT 池或自定义币种范围。
- 用户能配置随机搜索与智能搜索。
- 用户能配置多种间隔、金额、止盈、止损、指标组合。
- Worker 能执行 K 线海选与成交级精测。
- 系统能输出候选 Portfolio 排行榜。
- 系统能显示训练、验证、测试、压力场景表现。
- 系统能识别过拟合风险。
- 系统能显示完整风险摘要。
- 用户能确认发布 Portfolio 到实盘。
- 实盘启动前能检查 Hedge Mode、margin mode、leverage、symbol 冲突。
- 运行中能暂停/恢复/停止 Portfolio 或单策略。
- 异常恢复能识别 orphan order 并保守暂停。
- 现有 ordinary/classic grid 行为不回归。

## 20. 已知风险与处理

### 20.1 马丁尾部风险

马丁策略在长期单边行情中可能快速扩大亏损。处理方式是强制预算、止损、强平风险、压力场景和生存优先评分。

### 20.2 回测过拟合

随机/智能搜索容易过拟合历史数据。处理方式是训练/验证/测试切分、walk-forward、压力场景和过拟合标记。

### 20.3 数据源不可访问

当前外部 SQLite 文件存在但命令行打开失败。处理方式是实现只读探测、明确诊断、支持配置只读副本路径。

### 20.4 同 symbol 多策略统计混乱

处理方式是引入 strategy instance、cycle、leg 归因，统计分为 strategy、symbol、global 三层。

### 20.5 合约账户设置冲突

处理方式是同 symbol 强制统一 margin mode 与 leverage，发布前检查持仓和挂单，不静默切换。
